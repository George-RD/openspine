# Design: Kernel briefcase packing

`openspine-schemas::briefcase::pack` remains pure and canonical. `openspine-kernel::briefcase` owns the live boundary: it projects `TaskGrant` to stable semantic fields (including `allowed_egress_classes`), resolves the route/workflow/counterparty shape, selects relevant source slices from the kernel-owned `learned_sources` table, and persists the resulting JSON blackboard in `briefcases` keyed by grant id.

`CounterpartyRef` is an enum: `Bound { identity_id, relationship }` for identity-store-bound counterparties, or `Unresolved { channel, identifier }` when no binding exists. Unresolved counterparties carry only an opaque digest (`email:<hex>`), never the raw plaintext address (D-012). Unresolved counterparties map to the Stranger tier. This prevents masquerading a non-identity Ulid (e.g. event id) as an identity.

Relationship kinds coarsen to Stranger, Known, Intimate, or Owner. The static depth table is the product of tier weight and task-class weight. Depth limits the number of relevant preferences and skills packed (not just metadata). Current pipeline lanes derive Conversation for owner control and DraftApproval for selected-thread external communication. This derivation is candidate implementation canon.

The Grant→Run boundary binds the selection token before packing, then attempts to roll it back if packing or atomic grant+briefcase persistence fails. Packing failures fail closed and abort the task before the grant/briefcase rows are written; rollback failure is surfaced rather than ignored. The HMAC key check runs before grant binding so no selection token is intentionally left behind by the normal failure path.

The email lane performs a **bounded, metadata-only** recipient read during Verify-stage preflight: `threads.get?format=minimal` is streamed with a 256 KiB response cap, its returned IDs are truncated to the newest 20 locally, then `messages.get?format=metadata&metadataHeaders=From` is streamed with the same cap for each ID (at most 20). This catalogued `resolve_email_counterparty` effect path (classified `PreGateOwnerSelectedRead`) is authorized by owner selection + containment guard, NOT the authority gate. The raw address stays request-local; only the digest is persisted. The body-full `threads.get?format=metadata` read is explicitly avoided so the full thread body never crosses the pre-gate boundary.

Top-up mediation resolves only sources relevant to the stored briefcase workflow and tier. The kernel computes the payload digest, fills `TopUpDecision.source_digest`, and applies through the schema replay/key guard. Duplicate request IDs are rejected before evaluation (denials included). `Store::mutate_briefcase_and_audit` wraps load/mutate/write/audit in a `BEGIN IMMEDIATE` transaction, so two independent connections cannot both pass the replay guard and overwrite each other. The audit kind derives from outcome: `briefcase.topup.applied` for allowances, `briefcase.topup.denied` for denials — never unconditionally `applied`. Justification text is capped at 200 chars and persisted only as a digest via `TopUpRequest::for_persistence()`; section keys are capped at 128 bytes before gate evaluation and the request's persisted `section_key` is `key:sha256:<hex>`, while trusted source section keys remain canonical source identifiers. The raw request stays request-local. `TopUpPolicy` carries both a per-kind `max_depth` (governs one source's disclosure depth) and a separate `max_total_sections` ceiling (governs aggregate growth beyond initial depth). Missing `max_total_sections` fails closed to `self.depth`.

Worker visibility is persisted per `(task_grant_id, worker_id)`. Two task-token-authenticated endpoints form the production boundary: `GET /v1/briefcase` projects the filtered view, and `POST /v1/briefcase/:id/topup` mediates top-up requests (both allowed and denied decisions are durably recorded). The `POST /v1/briefcase/export` endpoint is **deferred** to `implement-worker-runtime`: the schema-level negative visibility guard (`export_returned_output` refuses non-`ReturnedOutput` keys) is tested in `openspine-schemas`, but this change ships no kernel-mediated producer because no worker-side `ReturnedOutput` section producer exists yet.

Gmail connector errors carry only `{ status: Option<u16>, class: GmailFailureClass }` — never the provider response body. The preflight failure and audit log record only status + fixed class label.
